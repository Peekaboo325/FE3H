# 개발기록 (DEVLOG) — FE3H v2.0

> **이 문서의 목적**: Claude Code 컨텍스트를 비워도 개발을 이어가기 위한 '살아있는 기록'.
> 새 세션은 CLAUDE.md → 이 문서를 읽고 현재 상태를 복원한다. **의미 있는 변경 후엔 이 문서를 갱신한다.**
> 최종 갱신: 2026-06-06 (인물 뷰/에디트·히어로, 모바일 마감, 사이트 껍데기, Vercel 배포 교훈)

## 1. 한눈에 — 지금 무엇이 돌아가나

루트에 v2.0 인터랙티브 픽션 앱이 서 있고, Vercel에 배포되어 실제로 동작한다.

핵심 고리:
```
[화면(React)] → [서버 한 겹] → [Claude Opus 4.8] → 본문 스트리밍 → [Supabase 저장] → 화면 복원
                   └ 주입: 박제 세계관(캐싱) + 큰 줄기·최근 화별 요약 + 활성 인물·문헌 + (지목 시)앵커
```

- **서버 한 겹**: 로컬은 Express(`server/index.mjs`), 배포는 Vercel 서버리스 함수(`api/*.mjs`). 둘이 같은 일.
- **원본 저장소 = Supabase**(클라우드). 브라우저는 Supabase에 직접 접근하지 않고 항상 서버(/api)를 거친다.
- **모델 분배**: 메인 본문 = Claude Opus 4.8(effort 'low', adaptive thinking). **Gemini(gemini-2.5-flash) = 화별 요약·큰 줄기·앵커 추림·인물 분석 보고서에 사용 중**(`lib/summarize.mjs`·`lib/report.mjs`). 미연결: 편지 7종·인물 추출·해시태그.

## 2. 기술 스택 / 운영

- Vite + React 19 + TypeScript. (Zustand 미도입 — 아직 상태가 단순)
- 로컬 개발: `npm run dev` (vite 5173 + express 8787, vite가 `/api`를 8787로 프록시).
- 빌드: `npm run build` (= `npm run bake && vite build`).
- **세계관 굽기**: `npm run bake` → `scripts/bake-worldview.mjs`가 `worldview/` 박제 4종을 `lib/worldview.mjs`로 구워 넣음(dev·build 시 자동). **worldview/ 고치면 재배포해야 반영.** (옛 `md/` 폴더는 worldview/로 개칭; 안 구워지던 참고 템플릿 3종은 삭제 — 인물·견문록은 유저 커스텀 DB로만 관리.)
- **배포 = `main`에 push** → Vercel 자동 배포(프로덕션 도메인 `crest-if.vercel.app`). 개발 브랜치 `claude/wizardly-dirac-WjCg3`에서 작업 → `main`에 머지 → push.
- 배포 전 항상 `npm run lint`(tsc) + `npm run build` 통과 확인.
- ⚠️ **Vercel 자동배포가 불안정한 진짜 원인(중요)**: Claude Code 웹 세션은 **클라우드 샌드박스 → git 프록시 → GitHub** 경로로 push한다. 커밋은 GitHub에 **항상** 도달하지만(매번 MCP로 확인됨), 이 경로의 push는 GitHub→Vercel **자동 웹훅을 들쭉날쭉하게만** 깨운다(어떤 푸시는 배포, 어떤 푸시는 무시). **한도/권한 문제 아님**(Usage 여유, 권한 정상). GitHub이 **직접** 내는 push(웹 편집·본인 머신)는 항상 정상 트리거됨.
- ⚠️ **Deploy Hook도 막다른 길이었음**: GitHub Action이 Deploy Hook을 정상 호출(초록 성공)해도 Vercel이 배포를 안 만듦(이 프로젝트의 git연동/Hook 처리 자체가 불안정). 즉 Vercel의 트리거 수신부를 의존하는 방식은 전부 실패.
- ✅ **최종 해결 = GitHub Action에서 Vercel CLI로 직접 빌드·업로드**: `.github/workflows/deploy.yml` — `on: push(main)` + `workflow_dispatch`. GitHub 러너가 `vercel pull`→`vercel build --prod`→`vercel deploy --prebuilt --prod` 로 **결과물을 직접 프로덕션에 올림**(Vercel git연동/Hook 안 거침 → 그쪽 불안정과 무관). Action 트리거는 샌드박스 push에도 안정적(실측 3/3 실행 확인). **비밀값**: `VERCEL_TOKEN`(Vercel Settings→Tokens), `VERCEL_ORG_ID`(Settings→General). `VERCEL_PROJECT_ID`(=`prj_pO0MqjnDbLnnOH7coic2GnjiU7FT`)는 비밀 아니라 워크플로 env에 박음. 자동 누락 시 **Actions→Run workflow** 수동.
- 옛 `VERCEL_DEPLOY_HOOK` 시크릿은 미사용(삭제 가능). 코드는 평소대로 `main` 머지+push만 하면 Action이 빌드·배포까지 끝냄.

### 환경 변수 (Vercel 설정 + 로컬 `.env`)
- `ANTHROPIC_API_KEY` — 클로드 본문. **(필수)**
- `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` — 저장소. **service_role(비밀) 사용. anon 아님.** (없으면 저장 없이 동작)
- `GEMINI_API_KEY` — 설정만 됨, 아직 쓰는 코드 없음.
- ⚠️ Vite는 `VITE_` 접두사 변수를 브라우저에 노출 → 비밀 키에 절대 `VITE_` 붙이지 말 것.

### Supabase 표 (SQL Editor에서 1회 실행, RLS on·정책 없음 → service_role만 접근)
- `stories(id, title, created_at, updated_at)` — 세이브 슬롯(이야기).
- `turns(id, role, content, story_id→stories on delete cascade, created_at)` — 본문(이야기별).
- `characters(..., name, english_name, aliases, base, gender, faction, rank, crest, appearance(미사용), height, build, hair, iris, impression, personality, combat, notes, bonds(jsonb), life_status, is_active, thumbnail, avatar, sort_order, title(미사용), story_id→stories cascade, created_at)` — **이야기별 분리**. `thumbnail`=초상(뷰 히어로용), `avatar`=얼굴(명부 목록 둥근 썸네일·없으면 thumbnail). 약력 항목: 신원(성별/소속/신분/문장/상태), 용모(신장·체격·모발·홍채·인상), 성향·전법·비고(마크다운), 인연(bonds). `appearance`·`title`은 옛 필드(미사용·DB 잔존).
- `lore(id, title, category, body, is_active, sort_order, story_id→stories cascade, created_at)` — 견문록(=대륙 문헌), **이야기별 분리**.
- ⚠️ **요약/연대 문헌용 추가 컬럼 (실기 전 1회 실행 필요 — 빌더가 아직 안 돌렸으면 폴백 모드로 동작)**:
  `alter table turns add column if not exists summary text;`
  `alter table stories add column if not exists summary text;`  (큰 줄기)
  `alter table stories add column if not exists summary_turn_id bigint;`  (흡수 포인터)
  (셋 다 없어도 안 깨짐 → '전체 원문' 폴백. 넣어야 화별 요약·연대 문헌·큰 줄기 작동.)
- ⚠️ **인물 명부 확장 컬럼 (약력 개편 — 각 1회 실행. 없으면 그 항목만 저장 안 됨)**:
  `alter table characters add column if not exists avatar text;` (목록 얼굴)
  `alter table characters add column if not exists rank text;` (신분)
  `alter table characters add column if not exists crest text;` (문장)
  `alter table characters add column if not exists gender text;` (성별)
  `alter table characters add column if not exists base text;` (거점)
  `alter table characters add column if not exists height text;` `... build text;` `... hair text;` `... iris text;` `... impression text;` (용모 5항목)
  `alter table characters add column if not exists bonds jsonb;` (인연 배열)
  `alter table characters add column if not exists analysis jsonb;` (분석 보고서 — 보고서 탭. 없으면 발급 시 "보고서 담을 자리 없음" 안내로 폴백)
  (※ 서버 코드(db.mjs 인물칸)도 바뀌므로 **로컬은 SQL 실행 후 dev 서버 재시작** 필요. 프로덕션은 같은 Supabase라 SQL 1회면 공유됨.)

### 진단
- `/api/turns`, `/api/characters`, `/api/lore`를 브라우저로 열면 `{dbReady, error, ...}`가 보임.
  - `dbReady:false` → 키 미설정/이름오타/미배포. `error`에 "relation ... does not exist" → 표 없음. "row-level security/permission" → anon 키 잘못 사용.

## 3. 데이터 분류 (박제 / 커스텀) — 핵심 설계

| 층 | 내용 | 처리 |
|---|---|---|
| **박제** (원작 정사·작법) | 세계관 규약 · 집필 지침 · 성인향 묘사 지침 · 고유명사 사전 | `worldview/` → bake → 시스템 프롬프트로 **항상 주입 + 프롬프트 캐싱** |
| **커스텀 — 인물** | 인물 설정 사전 · 주요 인물 노트 | **인물 기능**(characters 표 + UI) |
| **커스텀 — 세계** | 연재 고유 설정 | **견문록 기능**(lore 표 + UI) |

- 새 고유명사는 고유명사 사전(박제)을 건드리지 않고 커스텀(인물/견문록)에 추가한다.

## 5. 주요 결정과 이유 (왜 이렇게 했나)

- **md(문서) vs .ts(코드)**: 세계관을 .md로 두는 건 클로드 입장에선 동일(텍스트만 봄). 단지 비개발자가 편집하기 쉽고, 콘텐츠(유저 영역)와 코드(시스템) 분리. 코드가 직접 import 못 하니 bake로 다리를 놓음.
- **Supabase = 창고(DB)**, Vercel = 앱·서버. Supabase를 '우리 서버'로 쓰지 않고 서버 뒤 창고로 둠(키 숨김).
- **브라우저 캐시 = IndexedDB**(localStorage 5MB·동기 한계 회피). 단 원본은 Supabase, 캐시는 임시 사본.
- **썸네일 = WebP**(wbmp 아님 — wbmp는 흑백 1비트 구식). 600px 리사이즈. DB 인라인 저장(소량이라 Storage 버킷 생략).
- **인물 주입은 '전부'가 아니라 핵심+참조라벨**: 다 넣으면 모델이 외양·무기를 습관적으로 읊어 글이 산만 → 집필 지침의 "아는 걸 다 꺼내지 말 것"과 충돌.

- **본문 문체**: 박제 프롬프트가 Gemini Flash 기준이라 클로드용 톤 재조정 여지. 실제 출력 보고 조정.
- **로딩 애니메이션 + 앵커 확인 자취 — 적용 완료**: `src/Loading.tsx`(점 3개 bounce + 디제틱 문구 4초 페이드 교체, 일반/회상 두 묶음). `…집필 중…` 대체. **회상 톤**: 입력에 앵커 있으면(`src/anchorDetect.ts` 클라 감지) 로딩이 보랏빛·회상 문구로(즉각 확인). **확인 자취**: 서버가 *실제로 되짚은* 회차·문헌을 응답 헤더 `x-recall`(URI 인코딩 JSON)로 보냄 → `anchor.mjs`의 build*Context가 `{block, episodes/items}` 반환하게 변경, story.mjs·server가 본문 전 setHeader. 클라(App)가 그 헤더를 읽어 **토스트**(`.recall-toast`, 위에서 떴다 3.2s 후 사라짐)로 "❧ 되짚은 기록 — 제3화 · 제2권 퍼거스 신성 왕국"을 잠깐 알림(디제틱·시스템어 0, 본문/DB 무관·화면 표시뿐). regen은 앵커 미적용이라 일반 로딩·토스트 없음. 못 찾으면 헤더·토스트 없음(헛걸음 방지).
- **렌더 성능(100화+ 대비) — (a) 간단 윈도잉 적용 완료**: `App.tsx`가 평소엔 최근 `WINDOW`(=30)칸만 렌더, 그 위는 '❧ 이전 두루마리 펼치기' 버튼(`.unfurl`)으로 `STEP`(=30)씩 펼친다. 이야기 새로 불러오면 `loadTurnsFor`가 `visibleCount`를 WINDOW로 리셋(최근 창으로 접음). **인덱스 보존**: `start=max(0,len-visibleCount)`로 잘라 그리되 `i=start+vi`로 원본 번호 유지 → 복사·새로받기·수정·삭제 대상이 안 어긋남. 자동스크롤(끝 ref)·스트리밍·읽기모드와 충돌 없음. ⚠️ 더 필요하면 (b) 진짜 가상 스크롤 `@tanstack/react-virtual`로 승격 — 단 스트리밍·자동스크롤·인라인편집과 상호작용 주의, 브라우저 없는 환경에선 실측 검증 어려움.
- **턴 개별 조작 완료**: 프롬프트/결과 ①수정 ②삭제 ③재생성. 마크다운 렌더·평문 복사 포함.
  · 수정/삭제: `/api/turns` POST·DELETE, App 인라인 편집+낙관적 업데이트.
  · 재생성(③): `api/regen.mjs`·server `/api/regen`(messages+story_id+turn_id) — 매칭 답변만 다시 씀(앞 대화+그 프롬프트만 컨텍스트, 뒤는 그대로). App `새로받기(turnIndex)`가 그 칸만 스트리밍 갱신, 실패 시 원문 복원.
  · 토대: loadTurns가 turn `id` 반환, 생성 후 reload로 id 확보.
- **이야기 복사 완료**: `copyStory(sourceId)`(lib/db) — 새 슬롯에 turns+characters+lore 통째 복사. `/api/stories` POST `{copy_from}` 처리, Stories 패널 행에 '복사' 버튼.

## 7. 작업 컨벤션 (이 프로젝트에서)

- 항상 **한국어**로, **비개발자**가 따라오게 설명(용어 풀어서).
- git add/commit/push 자동 처리. 위험한(되돌리기·삭제) 작업만 먼저 확인.
- 디제틱 위장(UI·메시지·코드 주석까지 포드라 안의 단어). UI 언어 사전은 CLAUDE.md 참조. legacy는 읽기 전용 참고.
- 배포는 main 머지 + push. 코드 변경 후 lint+build 통과 확인. 자동배포는 **GitHub Action**(`.github/workflows/deploy.yml`)이 Deploy Hook을 호출해 처리(위 §2). 샌드박스 push가 자동 트리거를 누락하면 **Actions 탭 → Run workflow** 또는 GitHub 웹에서 아무 커밋.
- **뷰/에디트 분리는 앱 전체 공통 원칙**: 모든 콘텐츠 기능(인물·문헌 등)은 읽기 뷰 → '편집'으로 진입. 새 패널 만들 때 이 패턴 따른다.
- 박제 콘텐츠(`worldview/` 4종)는 모델에 직접 박힘 → 표현 하나가 행동에 영향. 수정 후 `npm run bake`+재배포.
