# 개발기록 (DEVLOG) — FE3H v2.0

> **이 문서의 목적**: Claude Code 컨텍스트를 비워도 개발을 이어가기 위한 '살아있는 기록'.
> 새 세션은 CLAUDE.md → 이 문서를 읽고 현재 상태를 복원한다. **의미 있는 변경 후엔 이 문서를 갱신한다.**
> 최종 갱신: 2026-06-06 (컨텍스트 윈도우 적용)

---

## 1. 한눈에 — 지금 무엇이 돌아가나

루트에 v2.0 인터랙티브 픽션 앱이 서 있고, Vercel에 배포되어 실제로 동작한다.

핵심 고리:
```
[화면(React)] → [서버 한 겹] → [Claude Opus 4.8] → 본문 스트리밍 → [Supabase 저장] → 화면 복원
                   └ 주입: 박제 세계관(캐싱) + 활성 견문록 + 활성 인물
```

- **서버 한 겹**: 로컬은 Express(`server/index.mjs`), 배포는 Vercel 서버리스 함수(`api/*.mjs`). 둘이 같은 일.
- **원본 저장소 = Supabase**(클라우드). 브라우저는 Supabase에 직접 접근하지 않고 항상 서버(/api)를 거친다.
- **모델 분배**: 메인 본문 = Claude Opus 4.8(effort 'low', adaptive thinking). 나머지(편지·요약·추출 등) = Gemini(키만 설정, **아직 미사용**).

## 2. 기술 스택 / 운영

- Vite + React 19 + TypeScript. (Zustand 미도입 — 아직 상태가 단순)
- 로컬 개발: `npm run dev` (vite 5173 + express 8787, vite가 `/api`를 8787로 프록시).
- 빌드: `npm run build` (= `npm run bake && vite build`).
- **세계관 굽기**: `npm run bake` → `scripts/bake-worldview.mjs`가 `md/` 박제 4종을 `lib/worldview.mjs`로 구워 넣음(dev·build 시 자동). **md 고치면 재배포해야 반영.**
- **배포 = `main`에 push** → Vercel 자동 배포. 개발 브랜치 `claude/wizardly-dirac-WjCg3`에서 작업 → `main`에 fast-forward 머지 → push(양쪽).
- 배포 전 항상 `npm run lint`(tsc) + `npm run build` 통과 확인.

### 환경 변수 (Vercel 설정 + 로컬 `.env`)
- `ANTHROPIC_API_KEY` — 클로드 본문. **(필수)**
- `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` — 저장소. **service_role(비밀) 사용. anon 아님.** (없으면 저장 없이 동작)
- `GEMINI_API_KEY` — 설정만 됨, 아직 쓰는 코드 없음.
- ⚠️ Vite는 `VITE_` 접두사 변수를 브라우저에 노출 → 비밀 키에 절대 `VITE_` 붙이지 말 것.

### Supabase 표 (SQL Editor에서 1회 실행, RLS on·정책 없음 → service_role만 접근)
- `stories(id, title, created_at, updated_at)` — 세이브 슬롯(이야기).
- `turns(id, role, content, story_id→stories on delete cascade, created_at)` — 본문(이야기별).
- `characters(..., is_active, thumbnail, sort_order, story_id→stories cascade, created_at)` — **이야기별 분리**.
- `lore(id, title, category, body, is_active, sort_order, story_id→stories cascade, created_at)` — 견문록, **이야기별 분리**.

### 진단
- `/api/turns`, `/api/characters`, `/api/lore`를 브라우저로 열면 `{dbReady, error, ...}`가 보임.
  - `dbReady:false` → 키 미설정/이름오타/미배포. `error`에 "relation ... does not exist" → 표 없음. "row-level security/permission" → anon 키 잘못 사용.

## 3. 데이터 분류 (박제 / 커스텀) — 핵심 설계

| 층 | 내용 | 처리 |
|---|---|---|
| **박제** (원작 정사·작법) | 세계관 규약 · 프로젝트 지침 · 성인향 묘사 지침 · 고유명사 사전 | `md/` → bake → 시스템 프롬프트로 **항상 주입 + 프롬프트 캐싱** |
| **커스텀 — 인물** | 인물 설정 사전 · 주요 인물 노트 | **인물 기능**(characters 표 + UI) |
| **커스텀 — 세계** | 연재 고유 설정 | **견문록 기능**(lore 표 + UI) |

- 새 고유명사는 고유명사 사전(박제)을 건드리지 않고 커스텀(인물/견문록)에 추가한다.

## 4. 빌드된 기능 + 파일 지도

- **이야기 생성·저장·복원**: `api/story.mjs`·`server`(스트리밍, turns 저장), `src/App.tsx`(켤 때 `/api/turns?story_id=`로 복원).
- **세이브 슬롯(이야기 관리)**: `stories` 표 + `api/stories.mjs`·`server`(목록/생성/이름변경/삭제). `src/Stories.tsx`(헤더 '이야기' 버튼) + App의 부팅 로직(현재 이야기 = localStorage `fe3h.currentStoryId`, 없으면 '이야기 1' 자동 생성). 본문 생성/저장은 현재 `story_id`로. 삭제 시 해당 본문 cascade. **인물·견문록도 이야기별 분리** — 각 패널의 "다른 이야기에서 불러오기"(`src/ImportDialog.tsx`)로 독립 복사. 인물·견문록 캐시/조회/주입 모두 `story_id` 스코프.
- **세계관 주입**: `lib/worldview.mjs`(박제, 캐시 블록) + `lib/charContext.mjs`(활성 인물) + `lib/loreContext.mjs`(활성 견문록). 인물·견문록 블록은 **박제 뒤에 붙여 캐싱 유지**.
  - 인물 주입 정책 = **B(핵심+참조라벨)**: 이름·이명·소속·정의·성격은 늘, 외양·무기·비고는 '참조용 — 필요할 때만' 라벨.
  - 견문록 주입: '원작 기본값보다 우선', '설명조 나열 금지'.
- **인물 프로필**: `src/Characters.tsx` + `src/imageUtils.ts`(업로드 이미지를 600px·WebP로 변환, 인라인 저장) + `lib/db.mjs` CRUD. 헤더 '인물' 버튼.
- **견문록**: `src/Lore.tsx` + `lib/db.mjs` CRUD. 헤더 '견문록' 버튼.
- **목록 캐시**: `src/useCachedList.ts`(stale-while-revalidate: 메모리 + IndexedDB) → `useCharacters`·`useLore`가 얇은 래퍼. `src/idbCache.ts`(의존성 없는 IndexedDB KV). **원본은 Supabase, 캐시는 빠른 사본.**
- **폰트**: 국문 SUIT(jsDelivr CDN @font-face), 영문 Hanken Grotesk(Google Fonts), 서명·명문 Pinyon Script.
- **본문 표현**: `src/StoryText.tsx`(react-markdown + remark-gfm)로 서술자 본문을 마크다운 렌더. 라틴(=포드라 문자)은 `src/podraScript.ts`의 rehype 플러그인이 `.script`(Pinyon)로 감쌈. 결과값 '복사' 버튼은 `stripMarkdown`으로 평문 복사.
- **테마**: 다크(메인 `#2d2d3a` + 양피지 금빛 강조). 색은 `src/styles.css`의 `:root` CSS 변수에서 한 곳 관리.
- **메인 화면(읽기/집필 모드)**: 헤더 토글(localStorage `fe3h.mode`). 읽기 모드 = 프롬프트·입력창·액션 숨기고 본문만(소설처럼 크게). 디제틱: 장면 구분 `⁂`(rehype가 hr→`.scene-break`), 회차 제목 가운데, 은은한 비네팅. 프롬프트는 우측 칩(글자는 좌측), 액션 아이콘도 우측.
- **메인 레이아웃·입력**: `.page` 100dvh 셸 + `body{overflow:hidden}`로 헤더·입력바 고정. `.scroll`은 전체 폭(스크롤바 창 끝), 본문은 `.scroll-inner` 720 가운데. 헤더 버튼(MenuIcon/모드토글)은 창 양끝, 제목 가운데. 입력칸 자동높이(1줄 시작→`composeRef` 측정), 버튼 라벨 '전개'. 엔터: 데스크탑 전송/Shift+Enter 줄바꿈, 모바일(coarse 포인터) 줄바꿈, 한글 조합 중 무시. 메시지마다 복사/새로받기/수정/삭제(lucide 아이콘, hover). 스크롤바 전부 얇게, 포커스 파란 아웃라인 제거(금빛 테두리). ⚠️ lucide `Menu`는 드로어 `Menu`와 충돌하니 `MenuIcon`으로.

## 5. 주요 결정과 이유 (왜 이렇게 했나)

- **md(문서) vs .ts(코드)**: 세계관을 .md로 두는 건 클로드 입장에선 동일(텍스트만 봄). 단지 비개발자가 편집하기 쉽고, 콘텐츠(유저 영역)와 코드(시스템) 분리. 코드가 직접 import 못 하니 bake로 다리를 놓음.
- **Supabase = 창고(DB)**, Vercel = 앱·서버. Supabase를 '우리 서버'로 쓰지 않고 서버 뒤 창고로 둠(키 숨김).
- **브라우저 캐시 = IndexedDB**(localStorage 5MB·동기 한계 회피). 단 원본은 Supabase, 캐시는 임시 사본.
- **썸네일 = WebP**(wbmp 아님 — wbmp는 흑백 1비트 구식). 600px 리사이즈. DB 인라인 저장(소량이라 Storage 버킷 생략).
- **인물 주입은 '전부'가 아니라 핵심+참조라벨**: 다 넣으면 모델이 외양·무기를 습관적으로 읊어 글이 산만 → 프로젝트 지침의 "아는 걸 다 꺼내지 말 것"과 충돌.

## 6. 알려진 한계 · 다음 할 일

- **컨텍스트 윈도우 — 적용 완료**: 서버가 DB를 권위 삼아 **'줄거리 요약(시스템) + 최근 5턴(2화+새 입력) 원문'**으로 재구성해 클로드에 보냄. `lib/memory.mjs`(prepareConversation·buildSummaryBlock, `KEEP_TURNS=5`) + `lib/summarize.mjs`(Gemini Flash 요약) + `db.mjs`(getStoryMemory/setStoryMemory). story.mjs·server 양쪽에 적용. **스키마**: `stories.summary text`, `stories.summary_turn_id bigint`(요약 1회 SQL: `alter table stories add column if not exists summary text; alter table stories add column if not exists summary_turn_id bigint;`). **안전 폴백**: 요약이 옛 턴까지 못 닿으면(컬럼 미존재/Gemini 미설정·실패) 자르지 않고 전체 원문 전송 → 맥락 유실 0. **요약 갱신**: 요청 시작 때 필요한 만큼만(이미 최신이면 호출 안 함). ⚠️ 컬럼 없을 때는 매 턴 재요약(낭비) — SQL 넣으면 해소. ⚠️ 지연: 옛 턴이 밀려날 때 스트리밍 직전 Gemini 호출 ~1초(원하면 응답 후로 이동 가능). ⚠️ **regen(다시 받기)·copyStory는 아직 요약 미적용**(regen은 프롬프트까지 전체 전송, 복사본은 첫 플레이 때 백로그 1회 재요약).
- **앵커링/선택 주입 — 적용 완료(회차 단위)**: 유저가 본문 입력에 `"N화 참고"`/`"N화 참조"`(범위 `79~81화`, 다중 `79,80,81화` 지원)라 쓰면, 서버가 그 회차를 통째로가 아니라 **목적(문체/사건/대사/감정선/장면 대구)에 맞춰 요약·발췌·전문 중 하나로 추려(Flash) 주입**. `lib/anchor.mjs`(parseAnchors·buildAnchorContext, v2는 N번째 서술자 본문=N화) + `lib/summarize.mjs`(pickAnchorBrief, 공용 callGemini로 정리) + story.mjs·server 양쪽 주입(시스템 맨 끝=대화에 가장 가까이). UI: `App.tsx`가 본문마다 `.ep-no`("N화") 표식 — 앵커 기준점. **opt-in**이라 Flash 비용은 지목 시에만. Flash 실패/회차 없음이면 앵커 없이 진행(안전). legacy `contextOptimizer.ts` 계승하되 '무조건 전문' → '목적별 형태'로 한 수 더.
- **고유명사 사전 선택 주입 — 안 하기로 결정(2026-06-06)**: 사전(~17KB)은 **철자(정발 표기) 정답표**일 뿐 지식원이 아니다(캐릭터 이해는 Opus 학습 데이터에서 옴). 게다가 박제 SYSTEM에 들어가 **프롬프트 캐싱**을 타므로 매 턴 부담이 이미 낮다. 선택 주입으로 트리밍하면 **못 잡은 이름의 철자 흔들림** 위험만 생기고 절감은 작다(마진 < 리스크). → **사전은 통째로 유지.** 선택 주입의 실효는 동적·비캐시 payload(인물 상세=B정책, 견문록=활성토글+앵커)에서 이미 챙기고 있음. ⚠️ 재검토 시점: 사전이 훨씬 커지거나 캐시가 자주 깨질 때.
- **견문록 앵커 — 적용 완료**: `"제N장 참조"`(장 번호) 또는 `"(제목 참조)"`(괄호 안 제목)로 견문록 지목. `lib/anchor.mjs`(parseLoreAnchors·buildLoreAnchorContext, listLore의 sort_order·id 순 = 제N장, 제목은 느슨매칭). 견문록은 짧고 정제돼 Flash 없이 **항목 그대로 주입**(Gemini 없이도 동작) — 특히 '잠듦' 견문록을 그때만 되살리는 데 쓸모. UI: Lore 패널 각 항목 앞 `제N장` 표식(`.ep-no.jang`). 회차 블록보다 앞, 견문록 블록은 시스템에 push.
- **앵커링 다음 단계 후보**: ①다중 회차 시 '딥리콜'로 최근창 축소(legacy) ②regen에도 앵커 적용 ③지목 회차가 최근창 안이면 중복 생략 ④인물도 이름/번호 앵커.
- **Gemini 일부 연결**: **줄거리 요약**에 첫 사용(`lib/summarize.mjs`, REST fetch, `gemini-2.5-flash`). 아직 미연결: 편지 7종·인물 추출·해시태그.
- **편지/서신 기능**: legacy 7종. Pinyon Script `.script` 서명체가 여기서 진가 발휘.
- **연대기·기억 시스템**: legacy 참고.

- **본문 문체**: 박제 프롬프트가 Gemini Flash 기준이라 클로드용 톤 재조정 여지. 실제 출력 보고 조정.

- **턴 개별 조작 완료**: 프롬프트/결과 ①수정 ②삭제 ③재생성. 마크다운 렌더·평문 복사 포함.
  · 수정/삭제: `/api/turns` POST·DELETE, App 인라인 편집+낙관적 업데이트.
  · 재생성(③): `api/regen.mjs`·server `/api/regen`(messages+story_id+turn_id) — 매칭 답변만 다시 씀(앞 대화+그 프롬프트만 컨텍스트, 뒤는 그대로). App `새로받기(turnIndex)`가 그 칸만 스트리밍 갱신, 실패 시 원문 복원.
  · 토대: loadTurns가 turn `id` 반환, 생성 후 reload로 id 확보.
- **이야기 복사 완료**: `copyStory(sourceId)`(lib/db) — 새 슬롯에 turns+characters+lore 통째 복사. `/api/stories` POST `{copy_from}` 처리, Stories 패널 행에 '복사' 버튼.

## 7. 작업 컨벤션 (이 프로젝트에서)

- 항상 **한국어**로, **비개발자**가 따라오게 설명(용어 풀어서).

- 디제틱 위장(UI·메시지·코드 주석까지 포드라 안의 단어). legacy는 읽기 전용 참고.
- 배포는 main 머지. 코드 변경 후 lint+build 통과 확인.
