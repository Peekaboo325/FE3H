# 물자 조달 시스템 설계서 (v2.0)

> 2026-06-14, 빌더와의 설계 논의로 확정. **미구현 — 도면.** 착수 시 이 문서가 기준.
> 동기: 소지품 일러스트 **512종**이 인물당 3점씩만 쓰여 아까움 → 자산을 통째로 무대에 올리는 자리.
> 성격: **순수 재미(수집·선물) 콘텐츠.** 본문(전황) 비연동. 게임화(숫자·재고관리) 회피 — 그냥 구경→조달.

---

## 0. 한 줄 정의

**인물 명부 아래** 새 메뉴 **'물자 조달'.** 6개 점포에서 AI가 **입하**한 물건을, 골라서 **개별 인물의 소지품창으로 '조달'**(선물)한다. 점포 진열은 소지품의 아이콘 매칭 엔진을 *분류 한정 + 상점 말투*로 변주한 것.

## 1. 점포 6 + 분류 매핑

| 점포 | 끌어 쓰는 분류(`ITEM_ICON_GROUPS.cat`) | 대략 |
|---|---|---|
| **식료품점** | 음식·식재료 | 77 |
| **잡화점** | 생활·도구 + 문서·기록 | 166 |
| **의상실** | 의복·천 + 장신구·기념품 | 141 |
| **대장간** | 무기·군용품 | 37 |
| **채집상** | 약초·꽃 + 자연물 | 48 |
| **암시장** | 종교·주술물 + 증표·불명물 + 마물 소재 | 43 |

- 6점포가 512종을 빠짐없이 덮는다. 점포→분류 매핑은 `lib/supply.mjs` 상수(`SHOPS`)에 둔다.
- 각 점포의 아이콘 후보 = 그 분류들의 `label` 목록(소지품 `그림라벨`을 분류로 필터). 매칭·폴백·별칭은 소지품 것(`itemIcons.mjs`) 그대로.

## 2. 입하 (생성) — Flash 1콜/점포

- 점포를 열 때 재고가 없으면(첫 방문·소진 후) 또는 **'새로 입하'**를 누르면, Flash가 그 점포 분류에서만 **6점**을 새로 짓는다.
- **본문 안 읽음**(freeform·순수 재미) — 점포 분류 + 현재 진열만 입력. 맥락 조립 불필요(소지품보다 가벼움).
- 한 점 = `{ id, name, comment, icon }` — 소지품과 동형. **name**=구체적·evocative한 물건명("겉이 노릇한 보리빵"), **icon**=그 분류 라벨 중 가장 가까운 것(name≠label, 시각 근사), **comment**=점포 풍미 한 줄.
- 엔진 = `lib/items.mjs`의 `buildItemsPrompt`/`runItems` 변주(`lib/supply.mjs`): ①라벨 풀을 그 점포 분류로 한정 ②인물·맥락 제거 ③말투를 '상점 신상'으로(아래 §4) ④현재 진열과 dedup.
- ⚠️ Gemini에 SYSTEM·NAMES 안 넣음(소지품과 달리 보상 고유명사 풍미 불필요 — 물건은 일반명사). 물질문화 금지 몇 줄(동양 문방구·게임 용어·여신 외 신)만 유지.

## 3. 재고·지속

- **점포별 현 재고를 이야기별로 저장** — `settings` 표 `supply:{storyId}` → `{ grocery:[…], general:[…], attire:[…], smithy:[…], forager:[…], blackmarket:[…] }`. **SQL 불필요**(settings 표는 기록 지침이 이미 씀).
- 처음 열면 자동 입하. **'새로 입하'** = 그 점포 6점을 새로 지어 교체.
- **'조달'(주기)하면 그 물건은 재고에서 제거**(소진, 빌더 결정). 0이 되면 빈 진열 + 새로 입하 안내.
- 이야기 환원/삭제 시 그 `supply:{storyId}` 행도 정리(deleteStory/revert에 동반).

## 4. 조달 (인물에게 주기)

- 진열의 한 점 → **'조달'** → 인물 선택 모달(서신 수신 지정식 `modal--list`, 전체 인물) → 그 인물 `analysis.belongings`에 **append**(소지품과 같은 저장·아이콘·신규 id).
- **점포 풍미 한 줄을 그대로 가져간다**(인물 맥락 재생성 X — 가볍게, 빌더 결정). 즉 소지품에는 AI 탐색물 + 조달물이 한 주머니에 섞인다(순수 재미·의도).
- 재고에서 제거(§3). 토스트: "{퍼스트네임}에게 {물건}을 조달했습니다."
- 콜 없음(DB만) — 조달은 공짜.

## 5. 다양성 보장 (빌더 요구: "문구는 항상 다양")

- **한 입하 안**: 6점은 서로 다른 갈래 + 문구도 각기 다르게(소지품 dedup+다양성 규칙 계승 — 변주·자연 파편 자리채움 금지).
- **입하를 거듭해도**: 같은 물건 재등장은 허용(빵집엔 늘 빵)되 **문구는 매번 새로 생성**(과거 문구 미저장). 프롬프트에 **템플릿 금지**("갓 구운 ~, ~한 냄새가 난다" 같은 한 틀 반복 금지)·temperature 높임.
- across-입하 dedup은 안 함(상점은 재입고가 자연 — 소지품의 '누적 비중복'과 다른 점).

## 6. UI

- **메뉴**: 인물 명부 **아래** '物資 조달'(`src/App.tsx` 메뉴, 인물 명부 다음). 라벨 '물자 조달'.
- **점포 목록**: 6 점포 타일/행(이름 + 간단 결). `Modal` 또는 전용 패널.
- **점포 안(진열)**: 물건 카드 6(소지품 `ItemCard` 재사용 — 그림+이름+한 줄), 각 카드에 **'조달'** 동작, 하단에 **'새로 입하'**(소지품 탐색식 FAB/버튼·스피너).
- **조달 대상 선택**: 인물 선택 모달(전체 인물, 서신 수신 지정 UI 재사용).
- 빈 진열: "진열이 비었습니다 — 새로 입하하십시오" 안내.
- 표준 준수: Modal·Button·IconButton·toast·strings UI(§1). 동작어 **`UI.procure='조달'`** 등재.

## 7. 파일·엔진

- `lib/supply.mjs`: `SHOPS`(점포→분류 매핑) + `runRestock({storyId, shop})`(입하 6점·dedup·아이콘 매칭, items.mjs 변주) + `buildRestockPrompt`(상점 말투).
- `lib/db.mjs`: `getSupply(storyId)`/`setSupply(storyId, data)`(settings 표) + 조달은 기존 `setCharacterAnalysis`로 belongings append.
- `api/supply.mjs`·`server/index.mjs`: GET 재고 / POST 새로 입하(`{story_id, shop}`) / POST 조달(`{story_id, shop, item_id, character_id}` → belongings append + 재고 제거).
- `src/SupplyMenu.tsx`(또는 패널) + `App.tsx` 메뉴 장착 + `useCharacters`/`strings.ts(UI.procure)`.
- 모델 = **Gemini Flash, `lib/gemini.mjs` 경유**. 입하=1콜, 조달=DB만.

## 8. 착수 전 전제

- `settings` 표 **기존**(기록 지침이 씀) → **SQL 불필요.** `analysis` 컬럼도 기존(belongings).
- **CLAUDE.md §1** UI 동작어 '조달' 등재. (§3 승인 예외 **불필요** — 점포 생성에 SYSTEM·NAMES 안 넣음.)
- 실호출 검증 = 빌더 로컬 dev(Gemini). lint+build 통과.

## 9. 알려진 한계 · 다음

- 조달물이 소지품의 'AI가 인물을 드러내는' 결을 흐릴 수 있음(왕 주머니에 암시장 독병) — '순수 재미'로 수용(빌더). 거슬리면 조달물 표식/구분은 훗날.
- 본문 비연동(freeform) — 훗날 점포에 계절·전황 풍미를 입히고 싶으면 그때 본문 주입(지금은 비용·단순함 위해 뺌).
- 점포 풍미 문구가 인물 맥락과 안 맞을 수 있음(조달 시 재생성 안 하므로) — 수용. 필요해지면 조달 시 1콜로 재contextualize 옵션.
