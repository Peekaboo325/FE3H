## 0. 철학 — legacy의 교훈

- **legacy가 발작한 근본 원인**: "이 발신자가 이걸 알 만한가"(거리·시선 판별)를 가진 적 없는 신호로 기계가 풀려 함 — `[가-힣]{2,5}` 정규식 이름 매칭, 확률 땜빵(모브 40%·답장 스킵 40%·역할 반전 35%), 조건부 프롬프트 조각들. 한 문제 잡으면 다른 데 터지는 누더기가 됨.
- **결정적 실패 구조**: 발신자 설명(잘 쓴 인연 description)을 먹이면서 **최근 본문도 같이** 먹임 → 모델은 거리를 추론해 절제하지 않고 눈앞의 것을 그냥 씀. **모델의 자제심에 기대는 모든 방법은 실패한다. 안 먹인 것만 안 샌다.**
- **v2 원칙**: 판별과 생성을 분리한다. 줄거리 공급 on/off는 모델이 아니라 **코드가** 쥔다.

## 1. 저장 — 새 `letters` 표

- 이야기별 분리(story_id cascade), **영구 누적**(소각 전까지). RLS on·정책 없음(service_role만) — 기존 표들과 동일.
- **한 통 = 한 행.** 발신·수신 양쪽 인물의 함에서 같은 통이 보인다(복제 없음 → 수정·소각 정합).
- **답장 깊이 1**: 원문→답장 1회로 끝. 답장에의 답장 금지. `reply_to_id`로 강제 — ⓐ`reply_to_id`가 있는 통은 답장 대상 제외, ⓑ이미 답장 달린 원문도 제외.

```sql
create table letters (
  id bigint generated always as identity primary key,
  story_id bigint references stories(id) on delete cascade,
  sender_id bigint,            -- 등록 인물 id, 모브/子(미등록)는 null
  receiver_id bigint,          -- 위와 동일
  sender_name text,            -- 표시명(모브 직함·子 이름 포함, 등록 인물도 박제)
  recipient_name text,         -- 수신인 호칭
  type text,                   -- 7종: letter/note/official/invitation/petition/warning/will
  status text,                 -- sent / draft
  title text, content text, signature text,
  reply_to_id bigint,
  is_sealed boolean default true,
  created_at timestamptz default now()
);
```
(착수·배포 전 빌더가 SQL 1회 실행. 없으면 폴백 안내 — 기존 패턴.)

## 2. 발신자 자격·통신 단위

| 발신자 | 자격 | 비고 |
|---|---|---|
| 모브 NPC | 가능 | 직함으로만 존재, 명부 미등록 |
| 등록 인물(母) | **활성일 때만** | 잠든 인물은 수발신 모두 제외(본문 주입과 같은 결) |
| 잠든 등록 인물 | 예외적 가능 | **활성 인물의 子로 등록된 경우만** — 그 母와의 쌍 안에서만. 빈도·자격은 子 대우, **재료는 본 약력** |
| 순수 子(미등록 인연) | 가능 | **자기 母하고만**(쌍이 통신 단위, 그 안에서 양방향). 다른 인물의 함 후보 아님 — legacy virtualCandidates 폐기 |
| 실종(불명)·사망 | **발신 불가** | 유일 예외 = 유서(§6, 생전에 써둔 것의 사후 발견) |

**수신 제한**: 실종·사망자 앞 발신 불가 — **母 본인이 쓴 경우만 생성하되 보관함(draft)행**('부치지 못한 편지'). 그 외 조합(모브 등→사망자)은 처음부터 생성 안 함.

## 3. 지식 판별 — 누설 방지의 핵 (2축, 코드 결합)

- **판별 A(사건)**: 최근 본문의 사건마다 "일어난 자리 밖으로 퍼지는가?" — **공개/사적 2단**. 기준은 규칙으로(예시 금지·§3): 생사·범죄·정치·신분·전투·스캔들=공개 / 일상·사소한 상호작용=사적.
- **판별 B(발신자)**: "이 인물이 그 장면에 직접 등장/관여했나?" — **의미 기반**(호칭 변형 인식, 정규식 아님). 후보 묶음 호출.
- **결합(코드)**: 발신자가 사건을 안다 = **현장에 있었다 OR 사건이 공개다**.
- **먹이는 수위**: 현장 = 원문 / 부재+공개 = **소문 수위(요약)** / 부재+사적뿐 = 줄거리 0(관계·약력만).
- 검산 사례(빌더 제시): 메이드 일으켜줌(사적)+샤하드(부재)=차단 / 사형수 탈출(공개)+샤하드=소문으로 앎 / A 다침+B(현장)=앎.
- 판별은 생성과 **별도의 좁은 분류 호출**(Flash·묶음). 파급 눈금이 거칠다 싶으면 국지/지역/대륙 3단으로 승격 가능(일단 2단).

## 4. 위치·빈도 (중세 통신 질감 + 뜬금 편지 추가 방어)

- **위치 출처**: **거점(base) 우선** → 소속(faction) → 빈 칸이면 '가까움'(fail open) / 子는 description에서.
- 거리→발신 가중치: 같은 거점 > 같은 나라 > 이웃 나라 > 대륙 밖 > **유랑·불명(최저)**.
- **빈 칸 ≠ 유랑/불명**: 빈 칸 = 가까움 취급(침묵 방지). 거점에 "유랑"·"불명"이라 **적힌** 경우만 최저 빈도 — 침묵은 빌더가 적었을 때만.
- 소프트(확률)·거칠어도 저위험. 지식 판별(§3)과 **독립된 별도 겹** — 빈도는 '얼마나 자주', 판별은 '뭘 아는지'.
- legacy 확률 땜빵과 다른 점: 디제틱 명분(통신이 느린 세계)이 있는 **단일 일관 규칙**.

## 5. 모브 표기 — 열린 명단 + 표기 안정화

- **명단은 닫지 않는다**(누가 올지 모르는 재미 유지 — 빌더 확정).
- 규율(프롬프트): 외래어 금지·한자어 지향·역할은 **단순 보통명사 하나**·**위치/구역 수식 금지**(장소는 본문에, 이름엔 안 붙임 — '동관 침실 시녀' 차단).
- **표기 안정화 레지스트리**: 그 이야기에서 이미 나온 직함 목록을 주입 → 같은 역할 = 같은 표기. 새 역할만 자유. (출처는 letters 표의 모브 sender_name들 — 별도 저장 불필요할 듯, 착수 때 확정)
- 후처리: 별칭 접기(셰프→요리사·메이드→시녀) + 분실물 대장식 로그(소지품 패턴).
- **legacy 8자 제한 해제** — 위 규율이 길이를 대신 다스림.

## 6. 수발신 흐름

- **트리거**: 서신 함의 **수동 새로고침만**, 1회 **1통**. 쿨다운 없음(생성 중 버튼 비활성만 — 기존 표준).
- **답장 우선**: 안 답한 받은 편지가 있으면 답장 후보. legacy의 40% 스킵 주사위 폐기 — "답할 사이·상황인가"를 **모델 판단**으로.
- **침묵의 권리(꽃①)**: 성격·상황상 답이 부자연스러우면 빈 배열로 거부 가능. 침묵 시 그 새로고침은 신규 발신으로 넘어감(유저 빈손 방지).
- **역할 반전 40%**: 함 주인이 발신자가 됨(발신함 채우는 길). 빌더: "보낸 편지도 궁금하다" — legacy 35%보다 올림.
- **유서**: 사망자당 **전 세계 통틀어 1통**(legacy는 쌍마다 — 폐기). 인연 있는 함 중 **먼저 새로고침한 곳에서 발견**("유품에서 당신 앞으로 된 편지가"). 그 뒤 그 인물은 영원히 침묵. 생사 게이트의 유일한 정당 예외(생전 작성·사후 발견).

## 7. Draft 보관함 (꽃②)

- **경로 3개**: ①AI 자발적(차마 못 보낸 속마음) ②사망·실종자 앞 자동 draft ③**답장 draft**(답을 썼는데 못 부침).
- **소재**: 母의 함 보관함에 **본인이 못 부친 것만**. 남의 미발송 편지 열람 없음(논리 구멍).
- **운명**: 열람 or 소각뿐. '부치기' 없음 — 못 부친 채로 남는 게 정체성.
- **빈도**: 기계 통제 없음 — Gemini의 캐릭터 해석에 맡김(v1 실측: 디미트리는 많고 벨레스는 적었음 — 성격이 빈도를 만듦). 프롬프트엔 "성격·상황상 차마 부칠 수 없을 때만" 원칙만.
- **답장 draft는 답장 자리를 안 잡아먹음**: 세상에 안 나간 편지 — 원문은 여전히 미답신(나중에 진짜 답장 가능). 단 같은 원문에 draft는 1회만(가드).
- 침묵과의 결: 밖으로는 침묵, 안으로는 draft — 침묵의 이층 서사.

## 8. 입력 패키지 (생성 호출에 들어가는 것)

| # | 입력 | 조건 |
|---|---|---|
| 1 | 발신자 재료(약력 / 子=description / 잠든 등록자=본 약력) | 항상 |
| 2 | 수신자 재료(약력) | 항상 |
| 3 | 둘 사이 관계(인연 칸) | 항상 |
| 4 | **둘 사이 이전 편지 최근 5통**(쌍 한정 — 다른 쌍 왕래는 누설) | 있으면 |
| 5 | 줄거리 — §3 판별 결과로 3단(원문/소문 요약/0) | 조건부 |
| 6 | 답장이면 원문 전문 | 답장 시 |
| 7 | 모브 직함 레지스트리 | 모브 관여 시 |
| 8 | 최근 편지 유형들(반복 방지) | 있으면 |
| 9 | 용어·호칭 묶음(§9) | 항상 |

- **draft는 4번에서 제외** — 안 보낸 편지를 보낸 셈 치는 사고 방지.
- 박제 세계관(SYSTEM)·기록 지침은 **안 넣음**(§3 원칙 — 클로드 본문 전용).
- 이전 편지의 가치: **말투 계승**(legacy 검증 — 호칭·종결어미 이어짐) + 화제의 연속(왕래가 작은 연재가 됨).

## 9. 용어 보호 — B안 (승인 예외, 서신 한정)

- 주입: **NAMES**(한글 표기 명단 — 임무 것 재사용) + **호칭 체계**(폐하=황제·국왕 / 전하=왕세자·섭정 / 예하=대사교 / 각하=맹주·재상 등 — legacy 용어 사전에서 발췌) + **금지 몇 줄**(하느님·주님 금지→여신님, 게임 용어 금지).
- 세계관 지식·역사·관계 설명은 안 줌(약력·인연이 맡음). "지식은 안 주고 맞춤법과 예법만."
- **후처리 치환 병행**(퍼디아→페르디아류 — legacy applyStrictTerminology 계승) — 이중벽.
- CLAUDE.md §3에 임무처럼 '승인 예외(서신 한정)' 추가할 것.

## 10. UI

- **명칭(확정)**: **수신함 / 발신함 / 보관함**. 3구획 전환(인물 서신 탭 안).
- **봉인**: 안 읽은 편지 = 봉인 표식, 열람 시 해제(`is_sealed`).
- **목록 행**: 표식(봉인/수신/발신) + 제목 + 상대 이름(네임드=퍼스트네임, 모브=직함). `ListRow` 등 기존 자산 재사용.
- **상세 뷰**: 제목·발신/수신·본문·서명(우하단). 편집 유지 + 소각(confirmAsk). 뷰/에디트 분리 원칙.
- **서명(확정)**: `[한글 수식어] + [영문 이름(Pinyon Script .script)]`
  - 이름 형태는 AI 판단: `Dimitri Alexandre Blaiddyd` / `Dimitri` / `Dima` / `D.` (격식·친밀도)
  - 수식어는 반드시 한글: "왕세자 Dimitri" (O) / "Crown Prince Dimitri" (X)
  - **모브는 영문 레터링 없음** — 한글 직함만, 흘림체 없음(네임드 서명의 대비 효과).
  - 영문 이름 출처: 등록 인물=`english_name`·nameDict / 子=AI 로마자 판단.
- **생성 버튼**: 임무·소지품 문법(하단 아이콘+스피너). 토스트 동사는 서신용 신규(예: "당도") — `strings.ts` UI에 등재 후 사용(§1 체크리스트).

## 11. 구현 메모 (착수 때 지킬 것)

- 모델 = **Gemini Flash, `lib/gemini.mjs` 경유 필수**(직접 fetch 금지 — 재시도·차단기 자동).
- **few-shot 금지**(§3) — legacy 프롬프트의 긍정 예시(서명 샘플 등)는 규칙·금지로 재작성.
- sender/receiver **ID는 모델 출력 안 믿고 코드가 강제 주입**(legacy 계승).
- 서명 가드: 빙의 차단(타인 이름 서명 금지) 계승. 단 legacy의 영문→한글 치환은 **반전**(이름은 영문 유지, 수식어 한글 검사).
- 저장은 `analysis` jsonb가 아니라 letters 표 — 보고서 merge 보존 로직과 무관.
- UI 문구는 전부 `strings.ts` UI 경유, 모달/버튼/토스트 표준 3종 준수.

## 12. 실기 라운드 보정 (2026-06-13 — 본문 §과 충돌 시 이 절이 우선)

빌더 실기 테스트(로컬 dev)에서 검증·교정된 최종 규칙. 상세 경위는 개발기록 2026-06-13 항목.

- **흐름 확률(§6 보정)**: 답장 우선은 무조건이 아니라 **시도율 50%**(안 답한 편지가 쌓이면 새로고침이 전부 답장으로 빠져 신규·모브가 굶는다 — legacy '40% 스킵'의 부분 복권, 명분: 기아 방지). **모브 = 신규 발신의 고정 40%**(가중치 틈에 끼우면 명부가 클수록 묽어져서 주사위 선행). 역할 반전 40% 유지.
- **구금 게이트(§2·§3 보정)**: 판별에 confined 판정 추가(구금·포로·뇌옥·중상·의식불명). 구금자는 **발신·답장 불가**(답장은 침묵 처리), 수신·draft 대상은 가능. 함 주인이 구금이면 받기만 한다. 상태 추적이 아니라 매번 최근 기록을 읽으므로 석방도 자동 반영.
- **유서 보정(§6)**: ①시간 방화벽 확장 — 발신자뿐 아니라 **수신자의 사후 신분·칭호**(즉위 등)도 모름(쓰던 시절의 모습으로 부른다). ②**온도는 인물 기록이 정한다** — 다정함 강제 금지, 떳떳하지 못한 자의 유서엔 회피·이중성. ③**유서 없는 자**(`유서없는자` Set): 안젤마(패트리샤) — 배신자라 유서를 썼을 리 없고 죽음을 예상 못 함(빌더 규약). 소각해도 재발견 안 됨.
- **호칭 5규칙(§9 보정)**: 관계>신분(아버지는 아들 왕에게 '아들아') / 쓰는 시점 기준(안 오른 자리 경칭 금지) / 한 사람 두 호칭 겹침 금지 / **자기 경칭 금지**('폐하의 명'을 왕 본인이 ✕) / 한 통 안 경칭 일관(전하·폐하 오락가락 ✕).
- **모브 두 얼굴(§5·§10 보정)**: 모델이 **role(직함)과 signature(서명)를 분리** 출력. 목록·발신인 = role(간결한 보통명사+짧은 한정어 하나까지: '시장 상인' 결. 구역·건물 수식 금지). 서명 = 자유 — 처지·연륜이 묻어나는 한 줄("성 아래 여인숙을 운영하는 보잘것없는 백성 **드림**"), 맺음 경어 허용. '올림·드림' 정리는 목록명·수신인에만.
- **소문 입말화(§3 보정)**: 판별 gist는 기록 문장을 베끼지 말고 **입말로 닳게**(내부 지시·관청 용어·수치 뭉개기) — "불가역적인 고문 금지"가 시장 상인 입에서 나오던 사고 차단.
- **이전 서신의 쓰임(§8 보정)**: "결은 잇되 대화는 잇지 않는다" — 신규 발신은 답신투 금지(답장 감사 인사 ✕)·**기왕 다룬 화제 재탕 금지**(새 용건 강제). 말투·호칭·거리감 계승만.
- **후처리 보강(§11)**: 본문 클리너는 **개행 보존**(`본문정리` — `\s{2,}` 일괄 치환이 문단을 뭉개던 버그). 본문 끝 서명 줄 자동 제거. 영문 서명 폴백 2단(명부 english_name → **NAME_DICT**(bake가 한→영 사전을 lib/worldview.mjs에도 구움)). 표기교정에 람베르트→람베르. 자발 draft는 **함 주인 발신만**(남의 draft는 보일 함이 없어 고아가 됨).
- **운영**: Gemini 무료 티어 429 RPD는 재시도 생략·즉시 안내. 에러 문구 디제틱("먼 곳의 응답이 뜸합니다…"), 토스트엔 한국어-단문만 통과(기술 문자열 필터).
