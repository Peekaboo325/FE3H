## 0. 프로젝트 정체성

- 파이어 엠블렘 풍화설월(Fire Emblem: Three Houses) 기반 팬 제작 **인터랙티브 픽션 앱**.
- LLM이 서사를 생성하고, 유저가 입력으로 이야기를 이어간다.
- **이 프로젝트의 핵심 가치는 '세계관 몰입도'다.** 모든 기술적 결정은 이 기준에서 내린다.

## 1. 절대 설계 원칙 (변경 불가)

1. **콘텐츠는 확정된 사실이 아닌 맥락 중심으로, 각자의 시선으로만.**
2. **모든 것은 포드라 안에서** — UI·시스템 메시지·본문·코드 주석까지 포드라 세계 안의 단어로 위장한다.
3. **노동의 순간조차 포드라 안에서** — 컨텍스트 입력·로딩 등 시스템 행위도 세계 안의 행위로 디제틱하게 위장한다.

### UI 언어 사전 (시스템어 → 포드라어) — 규칙으로 따른다, 단어별로 패치하지 말 것
새 UI 문구를 쓸 때 아래 표의 평범한 시스템어가 떠오르면 **반드시** 포드라어로 바꿔 적는다. 빠뜨리면 안 된다.

| 시스템어 | 포드라어 |
|---|---|
| 삭제/지우기 | **소각** |
| 취소 | **취소** |
| 저장 (저장 중…) | **기록** (기록하는 중…) |
| 복사 | **필사** |
| 수정/편집 | **편집** |
| 불러오기·가져오기(import) | **반입** |
| 불러오는 중(로딩) | **펼치는 중** |
| 이야기/세이브 슬롯 | **운명의 장**(개별), **천각의 박동**(이야기 메뉴 전체) |
| 인물 메뉴/패널 | **인물 명부**(컬렉션), 개별은 **인물** |
| 견문록 메뉴/패널 | **대륙 문헌**(컬렉션), 개별은 **문헌**, 셈 단위 **제N권**(앵커 `제N권 참조`) |
| 다시 받기(재생성) | **재작성** |
| DB 미연결 안내 | **기록의 샘이 닿지 않아…** |

- 이 표는 닫힌 목록이 아니다. 새 시스템어가 생기면 같은 결로 포드라어를 만들고 이 표에 추가한다.
- 본문 작법 톤("…하십시오")과도 어울리게. 위험 동작(소각)은 두 번 누르기 또는 소각 다이얼로그로.
- 예외: 빌더용 셋업 진단(예: "안내된 SQL을 한 번 실행" 같은 표 생성 안내)은 실효성을 위해 기술어를 남겨둘 수 있다.

### 표현 규칙 (확정)
- **기울임(이탤릭)을 쓰지 않는다 — 절대 규칙.** 디자인 강조든 `<em>`/`*기울임*`이든 정상체로. (CSS `em, i { font-style: normal }`로 못박음.)

### 말투 규칙 (확정)
- **하십시오체/~니다 체만 쓴다.** `~세요·~어요·~아요·~해요·~주세요` 같은 해요체는 **금지**. (예: "없어요"→"없습니다", "실행해 주세요"→"실행하십시오", "적으세요"→"적으십시오".)
- **한자어 지향.** 같은 뜻이면 한자어를 택한다. 취소(○) / 물림(✕) 같은 순우리말 변형은 쓰지 않는다.

### 새 UI 문구 체크리스트 (대책 — 디제틱이 새지 않게)
새로운 화면 문구(버튼·라벨·안내·에러·placeholder·aria-label)를 쓸 때마다 **반드시** 아래를 확인한다:
1. **행위어는 `src/strings.ts`의 `UI`에서 꺼내 쓴다.** (필사·소각·기록·반입·편집·취소·닫기·개칭·전개·재작성 등) — 시스템어(복사·삭제…)를 직접 타이핑하지 않는다. 새 행위어는 `UI`에 먼저 추가하고 그걸 참조한다. → 한 단어가 두 곳에 흩어져 어긋나는 일을 구조적으로 막는다(과거 "복사/필사" 버그의 원인).
2. **문장 끝이 해요체/~세요인지** 본다 → 있으면 ~니다/~십시오로.
3. **시스템어가 섞였는지** 위 사전으로 대조 → 포드라어로 치환.
4. ⚠️ 서버(`lib/*.mjs`)는 `src/strings.ts`를 못 가져온다. 서버가 만드는 표시 문자열(예: 이야기 필사 제목 `(필사)`)은 직접 적되 **UI 단어와 일치**시키고 그 옆에 일치 유지 주석을 단다.

### 버튼 표준 (대책 — 매번 같은 실수 반복 안 하게)
- **새 버튼은 직접 `<button>`+CSS로 짜지 않는다.** 텍스트/동작 버튼 = **`src/Button.tsx`(`<Button>`)**, 아이콘만 = **`src/IconButton.tsx`**. 둘 중 하나를 쓴다.
- `Button`이 자동으로 보장: **크기 고정**(loading이어도 라벨 자리를 유지하고 스피너만 겹쳐 → 폭 절대 안 변함 — *버튼 크기 변동은 빌더가 가장 싫어함*), 알약 모양, `variant`(primary=금빛 인장 / secondary=차분). 로딩은 `loading` prop, 라벨은 §1대로 `UI`에서.
- 즉 "내용 바뀌는 버튼(텍스트↔로딩)"은 **고민할 것 없이 `<Button loading>`** — 크기 안 흔들린다. 새 변형이 필요하면 `Button`/`.btn`에 추가하고 그걸 쓴다(개별 버튼에 일회성 CSS 금지).

### 모달 표준 (대책 — '또 새 모달 만들기' 금지)
- **새 모달은 `.modal-bg`/`.modal-head`를 직접 복붙하지 않는다.** **`src/Modal.tsx`(`<Modal>`)**를 쓴다 — 백드롭(바깥 클릭=닫기)+박스+표준 헤더(제목·(뒤로 `onBack`)·X·아래 구분선)가 자동. 본문은 children(경고문·`.modal-body`·푸터). 변형은 `className`(`modal--list`/`--chronicle`/`--import`…).
- 이미 이관: 연대 문헌·기록 지침·천각의 박동·반입. **예외(자체 2단/히어로 레이아웃)**: 인물 명부(`has-hero`)·대륙 문헌(코덱스) — 이건 `Modal` 안 쓰고 자체 구조(그래도 헤더 결은 맞춘다).
- 즉 새 패널 만들 때 "모달 어떻게 짜지" 고민 없이 `<Modal title onClose>`부터. 헤더가 또 제각각 어긋날 일이 구조적으로 사라진다.
- **헤더 문법(확정)**: 왼쪽(`head-side`) = 그 화면의 동작·뒤로가기, 가운데 = 제목, 오른쪽(`head-actions`) = **닫기(X)뿐**. 동작 버튼을 오른쪽에 몰아 담지 않는다.

## 3. 모델 분배 (확정)

런타임 호출은 콘텐츠 성격에 따라 모델을 나눈다. **전부 Opus로 도배하지 않는다.**

| 용도 | 모델 | 이유 |
|---|---|---|
| **메인 서사 본문** | Claude Opus 4.8 | 문장력·voice 핵심. 유일하게 승격하는 곳 |
| 편지·서신 시스템 (7종) | Gemini Flash | v1.0에서 품질 검증됨 + Sonnet보다 쌈 |
| 인물 추출·유형 분류·줄거리 요약 | Gemini Flash / Flash-Lite | 문장력 불필요한 기계적 처리 |
| 해시태그 생성 | Gemini Flash | 한국어 오타쿠 용어 센스 필요 |

- **메인 본문 = Claude Opus 4.8** 호출 설정: 노력(effort) '낮음', extended thinking 켜되 짧게, 세계관 문서는 **텍스트로 직접 입력**(PDF/파일첨부 금지 — 토큰 약 2배 부풀고 느려짐).
- Sonnet은 쓰지 않는다 (수빈 기준선 아래, 그럴 거면 v1.0과 차이 없음).
- 비용은 모델 다운그레이드가 아니라 **입력 토큰 축소**로 푼다.
- **Gemini 호출엔 박제 세계관(SYSTEM)·고유명사 사전을 넣지 않는다.** 그건 클로드 본문 전용(풍화설월 지식 보강용)이다. Gemini(요약·큰 줄기·앵커 추림·인물 분석 보고서)는 **그 과제에 필요한 입력만** 받는다 — 새 Gemini 프롬프트를 짤 때 worldview/nameDict를 import하지 말 것. (`lib/summarize.mjs`·`lib/report.mjs` 둘 다 현재 db만 import.)
- **프롬프트에 '예시(few-shot)'를 어지간하면 넣지 않는다 — v1.0 고질병(절대 전제).** 긍정 예시(원하는 출력 샘플)는 모델이 그대로 베껴 결과가 좁아지고 톤이 갇힌다. 원하는 결과는 **예시가 아니라 규칙·설명·금지(네거티브) 조건**으로 유도한다. 정말 불가피할 때만, 그것도 다양성을 강제하는 장치와 함께. (구조·형식 정의나 '하지 말 것' 네거티브 예외는 최소한으로.)

## 4. 비용·성능 설계

- **프롬프트 캐싱**: 고정 세계관 문서·확정 본문에 적용 (반복 입력 최대 90% 절감).
- **고유명사 사전 선택 주입**: 전체를 매번 넣지 말고, 그 회차 등장인물의 표기만 주입.
- **컨텍스트 윈도우 관리** (회차 누적 대비): 전체 본문을 매번 넣지 않는다.
  - 모델 입력 = '지금까지 줄거리 요약 + 최근 1~2화 전문'.
  - 전체 본문은 **Supabase DB**에 저장하고, 호출 시 필요한 창(window)만 꺼낸다.
  - 줄거리 요약 갱신은 Gemini(빌드/경량 역할)가 담당.
- **출력 길이**: max_tokens는 폭주 방지용 안전벨트일 뿐. 길이는 프롬프트로 제어한다(한 화=한 장면).
- **스트리밍**: 본문 생성은 스트리밍으로 체감 대기 감소.
- **비용 상한**: Anthropic Console 워크스페이스별 spend limit으로 하드 캡.

## 5. 정합성 규칙

- **날짜·연표·산술은 LLM 추론에 맡기지 않는다.** 확정값으로 계산해 입력에 주입한다.
  - (예: "더스커의 비극 = 현재 기준 9년 전"을 모델이 계산하게 두면 틀린다. 답을 박아서 넣는다.)
- 표기(고유명사 철자)는 고유명사 사전이 절대 기준. 사전에 없는 새 고유명사는 임의 표기 금지.
- 원작에 없거나 미확정인 사건·설정은 지어내지 않는다. 공백은 유저 확인 또는 '제안'으로 처리.

## 6. 기술 스택

- v1.0(legacy): TypeScript / React / Zustand / Vercel / IndexedDB / Gemini API

## 7. 빌더 컨텍스트

- **재사용 우선 — 만들기 전에 먼저 찾는다(기본값).** 컴포넌트·스타일·패턴·유틸을 새로 쓰기 전에 `Glob/Grep`으로 "이미 있나? legacy엔 있나?"부터 확인하고, 있으면 그걸 쓰거나 거기에 보탠다. 없을 때만 만들되 **공용으로**(다음에 또 쓰게). 같은 프로덕트 안에서 자산이 따로 노는 것 = 가장 싫어하는 것. 기본 블록: `Modal`·`Button`·`IconButton`·`Spinner`·`ListRow`·`strings.ts(UI)`·`lib/gemini.mjs`(공용 Gemini).
- 개발자는 **코드를 직접 치지 않는다.** 설계·방향을 정하고 Claude Code에게 빌드를 위임한다.
- **항상 한국어로 설명한다.** 비개발자가 따라올 수 있게, 전문용어는 풀어서 쓴다.
- 코드 변경 시: 무엇을·왜 바꾸는지 평이하게 설명한다.
- **git 작업(add / commit / push 등)은 알아서 처리한다.** 커밋 메시지도 알아서 적절히 단다. 단, 이미 올라간 기록을 되돌리거나 지우는 등 위험한 작업은 먼저 확인한다.
- **개발 루프 = 로컬 미리보기(`npm run dev`)**: 작업 중 화면 확인은 로컬로 한다 — 배포 없이 저장 즉시 반영. **배포는 '화면 보는 수단'이 아니다.**
- **배포(Vercel)는 절약·묶음**: 의미 있는 단위(기능 완성·빌더 확인 요청)로 **모아서** `main`에 올린다 — 작업 브랜치 커밋 → `main` fast-forward 머지 → push → GitHub Action이 빌드·배포(§8). 자잘한 변경마다·디버깅 목적으로 배포하지 않는다(§9 배포 절약). 배포 전 lint+build 통과 확인은 필수. **예외(먼저 상의)**: 위험한 git 작업, 구조를 크게 바꾸는 변경, DB 스키마/환경변수 변경, 되돌리기 어려운 일.
- **말투·리듬(중요)**: 따뜻하고 차근차근, 비개발자가 겁먹지 않게. 변경은 로컬로 확인하거나, 배포한 경우 "확인해 주세요"로 마무리한다. 큰 작업·구조 변경 전에는 방향을 먼저 상의하고, 갈리는 결정거리는 표로 짚어 묻는다. 원인은 추측하지 말고 로그·사실부터 확인한 뒤 고친다(§9). 이모지는 가볍게.

## 8. 개발 연속성 (컨텍스트를 비워도 이어가기)

- **작업을 시작할 때 `docs/개발기록.md`(DEVLOG)를 먼저 읽는다.** 거기에 현재 아키텍처·빌드된 기능·파일 지도·주요 결정과 이유·알려진 한계·다음 할 일이 정리되어 있다.
- **의미 있는 변경(기능 추가·구조 변경·결정)을 한 뒤에는 `docs/개발기록.md`를 갱신한다.** 이 문서가 새 세션의 기억이다 — 최신이어야 가치가 있다.

### 운영 치트시트 (자주 안 바뀌는 핵심)
- v2.0 = 루트. Vite + React 19 + TS. 서버 한 겹: 로컬 `server/index.mjs`(Express), 배포 `api/*.mjs`(Vercel 함수).
- 명령: `npm run dev`(로컬), `npm run build`(=bake+vite), `npm run lint`(tsc), `npm run bake`(worldview/→`lib/worldview.mjs`).
- **개발 = 로컬 `npm run dev`로 보면서.** **배포 = `main`에 push** → GitHub Action(`.github/workflows/deploy.yml`)이 Vercel CLI로 빌드·배포(Vercel 자체 git 자동배포·Deploy Hook은 불안정해 안 씀 — DEVLOG §2). 작업은 `claude/wizardly-dirac-WjCg3`에서, **배포할 때만** `main`에 ff 머지+push(모아서·§9 절약). lint+build 통과 확인.

## 9. 진단·디버깅 규율 (추측 금지)

- **막히면 추측하지 말고 사실부터 본다.** 에러·실패가 나면 가설을 나열하기 전에 **로그·에러 메시지·상태 코드를 먼저 확보**한다. (GitHub Actions 로그, Vercel 배포 로그, 콘솔 출력 등 — 직접 끌어올 수 있으면 끌어온다.)
- **원인을 확정하기 전에 수정 시도를 반복하지 않는다.** 재시도·재배포·설정 변경을 연타하는 것은 디버깅이 아니라 노이즈다. 한 번에 하나씩, 원인 확인 후 움직인다.
- **불완전한 데이터로 단정하지 않는다.** "화면에 안 떠서 아니다" 같은 판단 금지. 보이는 지표가 전부가 아닐 수 있음을 전제하고, 해당 원인을 직접 확인할 수 있는 로그를 찾는다.
- 원인을 모르겠으면 "모르겠다, 로그를 봐야 한다"고 말한다. 그럴듯한 추측으로 빌더의 시간을 쓰지 않는다.
- **git·배포 작업은 절약 원칙을 지킨다.** add/commit/push는 알아서 처리하되, 배포를 유발하는 행위(push, 빈 커밋, 재트리거, 리디플로이)는 **꼭 필요할 때만, 변경을 모아서 한 번에** 한다. 디버깅 목적의 반복 배포는 금지한다 — 무료 배포 한도(예: Vercel 하루 100회)나 비용을 빌더가 모르는 사이에 소진시킬 수 있다.
- 커밋 메시지는 알아서 단다. 단, **이미 올라간 기록을 되돌리거나 지우는 작업, 그리고 배포 한도·비용에 영향을 줄 수 있는 반복 작업은 먼저 확인한다.**
