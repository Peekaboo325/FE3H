# CLAUDE.md — FE3H (풍화설월 인터랙티브 픽션 앱 v2.0)

이 파일은 Claude Code가 이 레포에서 작업할 때 항상 따르는 지침이다.

---

## 0. 프로젝트 정체성

- 파이어 엠블렘 풍화설월(Fire Emblem: Three Houses) 기반 팬 제작 **인터랙티브 픽션 앱**.
- LLM이 서사를 생성하고, 유저가 입력으로 이야기를 이어간다.
- **이 프로젝트의 핵심 가치는 '세계관 몰입도'다.** 모든 기술적 결정은 이 기준에서 내린다.

---

## 1. 절대 설계 원칙 (변경 불가)

1. **콘텐츠는 확정된 사실이 아닌 맥락 중심으로, 각자의 시선으로만.**
2. **모든 것은 포드라 안에서** — UI·시스템 메시지·본문·코드 주석까지 포드라 세계 안의 단어로 위장한다.
3. **노동의 순간조차 포드라 안에서** — 컨텍스트 입력·로딩 등 시스템 행위도 세계 안의 행위로 디제틱하게 위장한다.

---

## 2. legacy 폴더 규칙 (중요)

- `legacy/` 폴더에는 v1.0 전체 코드가 들어 있다 (Google AI Studio + Gemini Flash로 빌드된 버전).
- **`legacy/`는 읽기 전용 참고 자료다. 절대 수정하지 마라.**
- v2.0은 레포 루트에 **새로 짓는다.** legacy의 좋은 설계(서신 로직, 캐릭터 분석, 연표 시스템 등)는 참고하되, 코드를 그대로 복사하지 말고 v2.0 구조에 맞게 새로 작성한다.

---

## 3. 모델 분배 (확정)

런타임 호출은 콘텐츠 성격에 따라 모델을 나눈다. **전부 Opus로 도배하지 않는다.**

| 용도 | 모델 | 이유 |
|---|---|---|
| **메인 서사 본문** | Claude Opus 4.8 | 문장력·voice 핵심. 유일하게 승격하는 곳 |
| 편지·서신 시스템 (7종) | Gemini Flash | v1.0에서 품질 검증됨 + Sonnet보다 쌈 |
| 인물 추출·유형 분류·줄거리 요약 | Gemini Flash / Flash-Lite | 문장력 불필요한 기계적 처리 |
| 해시태그 생성 | Gemini Flash | 한국어 오타쿠 용어 센스 필요 |

- **메인 본문 = Claude Opus 4.8** 호출 설정: 노력(effort) '낮음', extended thinking 켜되 짧게, 세계관 문서는 **텍스트로 직접 입력**(PDF/파일첨부 금지 — 토큰 약 2배 부풀고 느려짐).
- Sonnet은 쓰지 않는다 (수빈 기준선 아래, 그럴 거면 v1.0과 차이 없음).
- 비용은 모델 다운그레이드가 아니라 **입력 토큰 축소**로 푼다.

---

## 4. 비용·성능 설계

- **프롬프트 캐싱**: 고정 세계관 문서·확정 본문에 적용 (반복 입력 최대 90% 절감).
- **고유명사 사전 선택 주입**: 전체를 매번 넣지 말고, 그 회차 등장인물의 표기만 주입.
- **컨텍스트 윈도우 관리** (회차 누적 대비): 전체 본문을 매번 넣지 않는다.
  - 모델 입력 = '지금까지 줄거리 요약 + 최근 1~2화 전문'.
  - 전체 본문은 **Supabase DB**에 저장하고, 호출 시 필요한 창(window)만 꺼낸다.
  - 줄거리 요약 갱신은 Gemini(빌드/경량 역할)가 담당.
- **출력 길이**: max_tokens는 폭주 방지용 안전벨트일 뿐. 길이는 프롬프트로 제어한다(한 화=한 장면).
- **스트리밍**: 본문 생성은 스트리밍으로 체감 대기 감소.
- **비용 상한**: Anthropic Console 워크스페이스별 spend limit으로 하드 캡.

---

## 5. 정합성 규칙

- **날짜·연표·산술은 LLM 추론에 맡기지 않는다.** 확정값으로 계산해 입력에 주입한다.
  - (예: "더스커의 비극 = 현재 기준 9년 전"을 모델이 계산하게 두면 틀린다. 답을 박아서 넣는다.)
- 표기(고유명사 철자)는 고유명사 사전이 절대 기준. 사전에 없는 새 고유명사는 임의 표기 금지.
- 원작에 없거나 미확정인 사건·설정은 지어내지 않는다. 공백은 유저 확인 또는 '제안'으로 처리.

---

## 6. 기술 스택

- v1.0(legacy): TypeScript / React / Zustand / Vercel / IndexedDB / Gemini API
- v2.0(신규): 위 베이스 + **Supabase**(본문·에셋 영구 저장), Claude API(메인 본문) + Gemini API(나머지)
- 에셋은 **사전 제작**해서 Supabase Storage에 둔다. 런타임에 이미지를 실시간 생성하지 않는다.

## 7. 빌더 컨텍스트

- 개발자는 **코드를 직접 치지 않는다.** 설계·방향을 정하고 Claude Code에게 빌드를 위임한다.
- **항상 한국어로 설명한다.** 비개발자가 따라올 수 있게, 전문용어는 풀어서 쓴다.
- 코드 변경 시: 무엇을·왜 바꾸는지 평이하게 설명한다.
- **git 작업(add / commit / push 등)은 알아서 처리한다.** 커밋 메시지도 알아서 적절히 단다. 단, 이미 올라간 기록을 되돌리거나 지우는 등 위험한 작업은 먼저 확인한다.
- **자동 배포(기본값)**: 어지간한 변경은 묻지 말고 끝까지 배포한다 — 작업 브랜치 커밋 → `main` fast-forward 머지 → 양쪽 push(=Vercel 자동 배포). 배포 전 lint+build 통과 확인은 필수. 빌더가 Vercel 주소를 새로고침하며 피드백하는 게 빠르기 때문. **예외(먼저 상의)**: 위험한 git 작업, 구조를 크게 바꾸는 변경, DB 스키마/환경변수 변경, 되돌리기 어려운 일.
- **말투·리듬(중요)**: 따뜻하고 차근차근, 비개발자가 겁먹지 않게. 변경하면 배포까지 하고 "확인해 주세요"로 마무리한다. 큰 작업·구조 변경 전에는 방향을 먼저 상의하고, 갈리는 결정거리는 표로 짚어 묻는다. 원인은 추측하지 말고 확인(파싱·로그 등)한 뒤 고친다. 이모지는 가볍게.

## 8. 개발 연속성 (컨텍스트를 비워도 이어가기)

- **작업을 시작할 때 `docs/개발기록.md`(DEVLOG)를 먼저 읽는다.** 거기에 현재 아키텍처·빌드된 기능·파일 지도·주요 결정과 이유·알려진 한계·다음 할 일이 정리되어 있다.
- **의미 있는 변경(기능 추가·구조 변경·결정)을 한 뒤에는 `docs/개발기록.md`를 갱신한다.** 이 문서가 새 세션의 기억이다 — 최신이어야 가치가 있다.

### 운영 치트시트 (자주 안 바뀌는 핵심)
- v2.0 = 루트. Vite + React 19 + TS. 서버 한 겹: 로컬 `server/index.mjs`(Express), 배포 `api/*.mjs`(Vercel 함수).
- 명령: `npm run dev`(로컬), `npm run build`(=bake+vite), `npm run lint`(tsc), `npm run bake`(md→`lib/worldview.mjs`).
- **배포 = `main`에 push** → Vercel 자동 배포. 작업 브랜치 `claude/wizardly-dirac-WjCg3` → `main` fast-forward 머지 → 양쪽 push. 배포 전 lint+build 통과 확인.
- 저장소 = Supabase(표: `turns` / `characters` / `lore`, RLS on·정책 없음). 브라우저는 직접 접근 안 함, 서버가 **service_role** 키로만.
- 환경변수: `ANTHROPIC_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`(필수), `GEMINI_API_KEY`(미사용). **비밀 키에 `VITE_` 금지.**
- 진단: `/api/turns`·`/api/characters`·`/api/lore` 열어 `{dbReady,error}` 확인.
